## 2. 实验设置

> 由于没有清晰给出 box 和 item 的单位，并且其中包含数值尺度不唯一的情况，我们按照常识对所有尺寸进行统一，统一到微米单位并取整，以适应我们的离散算法

- 订单抽样规模：`1000`
- 订单抽样方式：随机种子的蓄水池抽样
- HTML 样例导出规模：`10` 个有效订单
- 物品尺寸缩放规则：原始尺寸统一乘 `10000`，再四舍五入取整
- 箱型尺寸缩放规则：
    - 删除长、宽、高任一为 `0` 的行
    - 若 `length < 1` 或 `width < 1`，则视为单位为米，先整体乘 `100` 转为厘米
    - 之后所有箱型尺寸再统一乘 `100` 到微米并取整
- 物品打标签逻辑：
    - 当物品名称中包含“盒”，我们将其打标签为 box
    - 当物品名称中包含“礼袋”，我们将其打标签为 bag
    - 其余物品统一打标签为 product，视为普通商品
- 预处理逻辑：
    - 若订单中包含属性为 `bag` 的商品（礼袋），这些商品会被直接移除，因为礼袋大小通常不是装箱大小的瓶颈，并且东西可以塞在里面，所以我们可以认为它没有体积，不参与装箱计算
    - 我们认为一次装箱中只有一个礼盒，不然会分为两单，所以若订单中包含属性为 `box` 的商品（礼盒），且数量超过 `1` 个，则跳过该订单
    - 若订单中仅含 `1` 个 `box` 商品，尝试将其作为内箱，优先装入其他普通商品

## 3. CSV 字段说明

| 字段                   | 含义                          |
| ---------------------- | ----------------------------- |
| `orderId`              | `orders.csv` 中的订单标识     |
| `actualBoxId`          | 生产数据中实际使用的箱型编号  |
| `actualUtilization`    | `物品总体积 / 实际箱体积`     |
| `algorithmBoxId`       | 装箱算法选择的箱型编号        |
| `algorithmUtilization` | `物品总体积 / 算法所选箱体积` |
| `status`               | 处理状态，正常行为 `OK`       |

## 4. 实验处理流程

1. 按上述规则清洗 `item.csv` 和 `box.csv`，并将尺寸统一转换到整数空间。
2. 读取 `orders.csv`，按相邻且相同的 `order-id` 聚合成订单块。
3. 使用固定随机种子，从全部订单块中抽样 `1000` 个订单。
4. 对每个抽样订单汇总物品数量，并基于真实箱型计算实际利用率。
5. 按候选箱型体积从小到大遍历；对每个候选箱运行 3D 装箱算法，直到找到第一个可行箱。
6. 将实际利用率、算法利用率、算法所选箱型和处理状态写入结果 CSV。
7. 再从有效订单中随机抽取 `10` 个，分别导出“实际箱装载结果”和“算法箱装载结果”的 HTML。

## 5. 算法流程

1. 所有要装的物品组成一个 list，先用体积进行从大到小排序
2. 算法先把 list 中的 bag 删掉
3. 提取出 box，进行第一阶段装箱，从第一个能装下的物品开始，贪心地装尽可能多的物品
4. 删掉已经被 box 装下的物品，让 box 代表他们，进行最终选箱
5. 把所有箱子从小到大排列，输出第一个可以装下的箱子

## 6. 结果数据

- 结果 CSV 总行数：`995`
- 实际利用率平均值：`0.298331`
- 算法利用率平均值：`0.423109`
- 比值平均值（`algorithm / actual`）：`1.634132`
- 比值中位数（`algorithm / actual`）：`1.557692`
- 算法优于实际的订单数：`745`（占比 `0.749497`）
- 算法劣于实际的订单数：`68`（占比 `0.068410`）
- 两者相同的订单数：`181`（占比 `0.182093`）

## 7. 结果解读

- 在当前样本中，算法利用率平均值为 `0.423109`，高于实际箱利用率平均值 `0.298331`。
- 利用率比值 `algorithm / actual` 的平均值为 `1.634132`，说明算法所选箱型在平均意义上达到实际利用率的 `163.413247%`。
- 在 `994` 个有效订单中，有 `745` 个订单的算法利用率高于实际利用率。

### 7.1 为什么算法存在更坏的情况？

第一，是算法本身存在很多无法考虑到的情况，比如数据本身没有考虑到物品折叠等情况，比如 sample-orders 中的 TSHE2026030500288559 订单，其中有一个物品 G4473101 记录的尺寸是 16.2,15.2,0.6 (cm)，但是实际订单使用的箱子尺寸是 21,12,11.5 (cm)，显然在不折叠物品的情况下是无法装下这个物品的，我们猜测是工人把这个东西打卷装箱了，但是这种东西很难在算法阶段考虑，如果加个额外的算法部分，当物品的很薄的时候考虑打卷，又无法保证物品本身是允许被打卷折叠的，所以这部分的最佳解决方案是在录入数据阶段解决，录入数据的时候测试其真实装箱情况下的体积。

第二，存在原始数据的问题，比如订单 TSHD2026030800130388 的实际利用率都已经达到 1.38 了，我们算法算实际利用率的逻辑是通过物品的体积和除以箱子体积，显然这个数值是绝对不可能超过1的，但是这个订单超过了快一半箱子体积了。并且实际利用率超过 1 的订单还是有的（抽10000 个订单，其中有 14 个超过 1 的），也存在大量实际利用率为 0.9 以上的订单，我们认为这样一个 NP-Hard 的问题人脑是很难规划出这么高的利用率的，所以我们认为部分数据本身是存在问题的。

> 问题应该不出在我们的数据缩放阶段，因为我们的缩放都是基于统一单位到微米的目标进行的，乘的都是 10 的次方倍，如果该阶段有错误，结果肯定是差十倍或者 0.1 倍，但是 1.38 这样的值显然是原始数据的问题或者有我们没考虑到的问题。
