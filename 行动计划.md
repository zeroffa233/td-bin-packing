# 装箱程序整体设计行动计划

## 1. 前言

本计划用于指导当前三维装箱项目从“核心算法已完成”走向“符合接口规范的完整程序”。

后续所有分析、流程图、模块设计与开发顺序规划，必须以 `算法流程.md` 为关键事实依据。该文档的信息密度很高，包含了入参语义、调度规则、拼箱规则、错误码逻辑与出参结构，不能只抽取其中的局部算法描述，更不能在后续分析中以当前代码实现替代接口事实。

本阶段的目标不是立即开发，而是先把需求、程序流程、模块边界和开发顺序设计清楚，并将这些内容沉淀为可协同修改、可直接指导实现的文档。

## 2. 设计原则

- 以 `算法流程.md` 为唯一一手业务依据，当前代码仅作为“已有实现现状”的参考，不作为需求来源。
- 以“完整请求处理程序”为设计对象，而不是只围绕核心极点装箱算法做局部包装。
- 先冻结程序流程、模块职责、失败语义和出参逻辑，再进入具体开发。
- 流程图和分析文档都必须服务后续交付，因此要具备可读性、完整性和一致性。
- 当前阶段优先完成文档设计，不急于进入编码实现。

## 3. 阶段计划

### 阶段一：形成程序分析文档

产出文件：`分析.md`

本阶段要把 `算法流程.md` 中的接口说明、决策逻辑和约束条件转化为“程序视角”的完整分析，而不是继续用字段解释的方式重复原文。

必须完成的内容：

- 明确一次请求的完整生命周期：输入标准化、前置校验、调度编排、后置校验、结果组装。
- 梳理顶层入参之间的关联关系：
  - `items` 与 `order.itemCode`
  - `containers` 与箱型候选选择
  - `lclRules` 与多轮拼箱决策
  - `unpackJudge` 与错误码表达式
- 提炼所有影响算法与调度的关键事实：
  - `csQty` 取余
  - `allowDown` 姿态限制
  - `carrierService` 分流
  - `stores` 箱型可用性约束
  - `maxWeight` / `capacityRate` 后合法性校验
  - `quantity` / 多箱循环
  - 礼盒、正品、赠品、小样的业务语义
  - `lclRules` 的优先级决策过程
- 分析当前核心算法与接口规范之间的差距，列出必须新增的外围模块能力。
- 明确“当前已知但尚不明确”的点，至少包括：
  - `SAME_TYPE` 的业务定义
  - `None` 在入参中的具体表达方式
  - `msg` 的错误信息组织规则

硬性要求：

- `分析.md` 中的每个结论都要能追溯到 `算法流程.md` 的明确事实，或标记为“当前假设/待确认项”。
- 不允许把 `算法流程.md` 中的复杂逻辑简化成“后续再看”的黑箱，尤其是 `lclRules`、`carrierService` 和错误码计算。

### 阶段二：形成程序流程图与说明

产出文件：`流程图.md`

该部分属于后续需要交付的内容，必须重点对待。

采用“1 张主图 + 多张子图”的方式组织：

- 主流程图：描述一次请求从入参到出参的完整链路。
- 子图 1：输入标准化与前置校验。
- 子图 2：核心调度与多箱循环。
- 子图 3：`lclRules` 拼箱优先级决策。
- 子图 4：后置合法性校验与出参组装。

流程图必须覆盖的关键节点：

- `carrierService=0/1` 的分流。
- `csQty` 取余后散件数量如何进入算法。
- 礼盒逻辑与散件逻辑如何衔接。
- 候选箱如何按照 `(sequence, volume)` 顺序尝试。
- 多箱循环何时继续、何时停止。
- `lclRules` 何时命中、何时降级、何时失败。
- `unpackJudge`、`ifMultiContainers`、`code/msg` 如何计算。

配套说明要求：

- 每张 Mermaid 图下都要补充节点说明。
- 每个关键判断节点都要说明输入条件、输出结果和失败出口。
- 图中的业务语义必须与 `算法流程.md` 保持一致。

### 阶段三：分析核心算法应做的适配与扩展

产出文件：`核心算法适配与模块化设计.md`

本阶段不是马上重构代码，而是先定义“当前核心算法在完整程序中应该处于什么位置、承担什么职责、不承担什么职责”。

要明确的问题：

- 核心算法模块的标准输入输出应该是什么。
- 哪些逻辑必须放在核心算法外围处理：
  - `csQty`
  - `allowDown`
  - 礼盒/散件处理
  - 候选箱过滤
  - 多箱调度
  - `lclRules`
  - 后置合法性校验
  - 结果展开与回写
- 当前核心算法是否需要新增参数化能力，例如：
  - 姿态枚举裁剪
  - 箱型尝试策略配合
  - 未装载物品的稳定输出

同时要完成模块化规划，至少形成以下职责边界：

- 请求标准化器
- 商品展开/数量处理器
- 候选箱过滤器
- 核心装箱器
- 多箱编排器
- 拼箱规则编排器
- 后置校验器
- 出参组装器

要求：

- 模块划分必须建立在 `分析.md` 和 `流程图.md` 的共识上。
- 模块职责必须可实现、可测试、可替换。

### 阶段四：规划外围程序开发顺序

产出文件：`开发顺序.md`

在前三份文档达成共识后，规划具体开发顺序。

建议按以下顺序展开，并允许在细化后调整：

1. 完成请求标准化与单箱快递主流程。
2. 接入多箱循环调度。
3. 接入后置合法性校验。
4. 接入物流快捷路径。
5. 接入 `lclRules` 编排。
6. 完成统一出参与错误码逻辑。
7. 补全日志、可观测性、测试与交付内容。

每一阶段都必须说明：

- 本阶段目标
- 依赖前置条件
- 变更的模块范围
- 验收方式

### 阶段五：开发外围内容

在以上设计文档冻结后，才进入具体实现。

开发阶段应遵循以下原则：

- 先做闭环最小主流程，再做复杂规则。
- 每个阶段完成后都能被独立验证。
- 保持核心算法模块与外围调度模块解耦。

### 阶段六：测试与交付

在程序实现接近完成时，形成完整的测试与交付方案。

测试至少应覆盖：

- 单箱成功装载
- 多箱装载
- `csQty` 取余
- `allowDown` 限制
- `stores` 匹配
- `maxWeight` 超重
- `capacityRate` 超限
- `carrierService=1` 快捷路径
- `lclRules` 命中/降级/失败
- `unpackJudge` 错误码逻辑
- 非法输入与缺失关联
- 出参 `infos` 展开正确性

交付至少应包含：

- 程序说明文档
- 流程图文档
- 关键规则说明
- 测试结果与典型边界案例

## 4. 当前执行方式

从现在开始，先围绕以下顺序协同推进：

1. 先起草并打磨 `分析.md`
2. 再形成 `流程图.md`
3. 再形成 `核心算法适配与模块化设计.md`
4. 然后形成 `开发顺序.md`
5. 最后进入开发与测试阶段

如果后续讨论中发现某些结论无法直接从 `算法流程.md` 推导，必须在文档中显式标注为“待确认项”或“默认假设”，不能静默补全。
