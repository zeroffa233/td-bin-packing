# 程序分析

## 1. 文档定位

本文档用于把 `参数列表.md` 中的接口事实、约束条件和决策逻辑，转化为“完整程序视角”的分析结果。

本文档中的结论遵循以下原则：

- 业务事实以 `参数列表.md` 为准。
- 当前代码只用于说明“已经实现了什么”，不用于反推业务需求。
- 不能从 `参数列表.md` 直接推出的内容，统一标记为“待确认项”或“默认假设”。

## 2. 一次请求的目标

一次请求对应一个订单的装箱决策。程序的目标不是只运行一次核心算法，而是要完成以下完整工作：

1. 解析并标准化请求中的商品、箱型、订单明细、拼箱规则与多箱约束。
2. 按 `groupId` 把一次请求中的 `orderDetail` 拆成多个独立调度组。
3. 对每个 group，根据 `carrierService` 判断走快递主流程还是物流快捷路径。
4. 根据 `csQty`、礼盒语义、姿态限制、`stores` 约束等规则，生成真正进入核心算法的候选商品与候选箱型。
5. 在每个 group 内必要时多次调用核心装箱算法，把商品拆分到多个箱子。
6. 在运行时对候选箱装法执行重量和容量百分比合法性校验，不合法则换箱或进入规则失败逻辑。
7. 合并所有 group 的装箱结果，按接口规范输出 `infos`、`code`、`msg`。

因此，该程序本质上是一个“调度编排程序”，核心极点装箱算法只是其中一个能力模块。

## 3. 顶层入参与程序语义

### 3.1 `items`

`items` 提供的是商品主数据，不是订单数量。程序必须通过 `code -> order.itemCode` 建立商品信息与订单明细的关联。

与程序逻辑直接相关的字段如下：

- `code`
  - 商品主数据主键，关联 `order.itemCode`。
- `itemCategory1`
  - 明确影响调度逻辑。
  - 根据文档描述，至少包含 `LH / 正品 / 赠品 / 小样`。
  - 其语义不是普通标签，而是会影响“礼盒优先装正品、散件再入外箱”的二段装箱流程。
- `itemCategory2`
  - 当前只知道可能值含 `易碎`，但用途未定。
  - 当前阶段记为待确认项，不纳入已知主流程。
- `weight`
  - 参与装后运行时重量合法性校验。
- `length / width / height`
  - 参与核心装箱算法。
  - 文档已明确需要支持精度缩放。
- `csQty`
  - 明确是前置数量处理规则。
  - `order.quantity % csQty` 的余数才是进入算法调度的散件数量。
- `allowDown`
  - 明确影响姿态枚举：
    - `1` 表示允许任意摆放
    - `0` 表示不允许放倒，只允许保持“向上”
- `volume`
  - 文档明确标注为不参与算法逻辑，不能作为体积事实来源。

### 3.2 `containers`

`containers` 提供的是候选箱型池，而不是单次调用的唯一箱子。

与程序逻辑直接相关的字段如下：

- `code`
  - 箱型输出标识。
- `length / width / height`
  - 箱型尺寸事实。
- `maxWeight`
  - 装后运行时重量合法性校验阈值。
- `capacityRate`
  - 装后运行时体积合法性校验阈值。
  - 装箱过程中不参与空间裁剪，但装箱结束后必须校验。
- `sequence`
  - 箱型优先级。
  - 文档明确要求按 `(sequence, volume)` 的字典序升序尝试候选箱。
- `quantity`
  - 当前请求允许使用的箱型库存。
  - 因为一个请求可能装多个箱子，所以箱型库存会在编排过程中递减。
- `stores`
  - 箱型前置可用性约束。
  - 只有当 `orderDetail.stores == container.stores`，或两者之一为 `None` 时，该箱型才能用于该商品。

### 3.3 `order`

`order` 是一次请求真正要调度的订单明细集合。程序必须以它为主线展开商品数量、批次信息和最终出参。

与程序逻辑直接相关的字段如下：

- `itemCode`
  - 关联到 `items.code` 获取商品信息。
- `quantity`
  - 与 `items.csQty` 配合，决定进入调度的散件数量。
- `batch / lot / manufactureDate / buildCode / zoneCode`
  - 参与 `lclRules` 决策。
- `stores`
  - 约束可使用的候选箱型。
- `carrierService`
  - 请求级分流字段。
  - 文档已明确：所有 `orderDetail` 该字段相同，因此只需在第一条上判定一次。
- `groupId`
  - 调度分组字段。
  - 同一个 `groupId` 内运行完整算法主流程；不同 `groupId` 之间互不混装。
  - 一次请求可以包含多个 `groupId`，程序需要逐个 group 调度，最后合并结果。

### 3.4 `lclRules`

`lclRules` 不是普通配置，而是一个带优先级的“多轮商品分组调度策略”。

其影响不是“如何在单个箱子中摆放”，而是：

- 当前优先级下，应该把哪些商品先拿出来尝试装箱。
- 如果该商品子集无法命中“刚好装不下/刚好装得下”的规则，需要如何降级到下一优先级。
- 如果规则耗尽后仍无法满足条件，要如何回退到“直接把剩余全部装好，但状态码返回失败”。

因此，`lclRules` 属于核心算法外围的编排逻辑，且是最复杂的业务流程之一。

### 3.5 `unpackJudge`

`unpackJudge` 不影响程序是否运行多箱装箱，而影响最终的错误码表达。

文档已经明确：

- 无论 `unpackJudge` 是 `0` 还是 `1`，程序都要正常执行“允许多箱”的装箱计算。
- 程序结束后再根据 `ifMultiContainers` 和 `unpackJudge` 组合计算 `code`。

因此，`unpackJudge` 是结果决策层字段，不是调度开关字段。

## 4. 程序主流程拆解

从程序视角看，一次请求必须至少经过以下阶段。

### 4.1 输入标准化阶段

职责：

- 校验请求结构是否完整。
- 建立 `items.code -> item` 映射。
- 建立 `containers.code -> container` 映射。
- 建立 `orderDetail.itemCode -> item` 关联。
- 统一尺寸精度缩放。
- 对 `quantity` 执行 `csQty` 取余。
- 展开真正进入算法的散件商品列表。
- 为每个商品补齐调度属性：
  - 主类别
  - 姿态限制
  - 重量
  - 批次/区域信息
  - `stores`

输出：

- 标准化后的订单商品集合
- 标准化后的候选箱型集合
- 请求级调度上下文

### 4.2 `groupId` 分组阶段

职责：

- 按 `orderDetail.groupId` 对标准化后的调度商品分组。
- 为每个 group 构造独立的调度上下文。
- 确保后续礼盒预处理、候选箱过滤、`lclRules` 编排和多箱循环都只在当前 group 内执行。
- 每个 group 输出自己的合法装箱结果，最终由请求级结果组装器统一合并。

### 4.3 请求分流阶段

职责：

- 读取首条 `orderDetail.carrierService`。
- 在每个 group 内决定进入：
  - 快递主流程
  - 物流快捷路径（直接选符合 `stores` 的最小箱型）

### 4.4 商品预处理阶段

职责：

- 基于 `itemCategory1` 识别礼盒相关语义。
- 挑出当前 group 内所有 `LH`。
- 按正品优先、小样/赠品在后的顺序，将商品轮询分配到所有 `LH` 中。
- 分配后的 `LH` 作为外层箱子调度商品。
- 没有 `LH` 时，所有散件直接进入外层箱子调度。
- 将 `allowDown` 约束转换为核心算法可识别的姿态枚举限制。

注意：

- 这一步与当前实验代码中的 `box / bag / product` 预处理逻辑存在概念相似性，但接口事实来源不同，不能直接等同。

### 4.5 候选箱过滤阶段

职责：

- 按 `stores` 过滤不合法箱型。
- 维护箱型剩余数量。
- 按 `(sequence, volume)` 升序形成尝试顺序。

### 4.6 核心调度编排阶段

职责：

- 对快递路径：
  - `lclRules` 始终作为主编排器。
  - 按优先级执行“商品子集选择 -> 当前规则尝试装箱 -> 命中/同规则扩子集/降级/失败回退”。
- 对物流路径：
  - 直接返回符合 `stores` 的最小箱型。

关键点：

- 多箱装载是编排层行为，不是核心算法本身的行为。
- 同一请求会多次调用核心算法。
- 同一请求内的不同 `groupId` 彼此独立调度，不应跨 group 混装商品。
- 每次调用都要更新：
  - 剩余商品
  - 剩余箱型数量
  - 已确定装箱结果

### 4.7 运行时合法性校验阶段

职责：

- 对每个候选箱装法在锁箱前做：
  - 重量校验：`maxWeight`
  - 容量校验：`capacityRate`
- 若不合法：
  - 更换候选箱型重新调度，或
  - 判定当前规则失败并进入降级/失败逻辑

注意：

- 该阶段不是最终汇总时的“临时报错”。
- 只有已经通过运行时合法性校验的箱子装法，才允许进入最终装箱结果集合。

### 4.8 出参与状态码组装阶段

职责：

- 合并所有 `groupId` 的合法装箱结果。
- 将合并后的装箱结果展开为 `infos` 数组。
- 保留订单原字段中要求原样透传的内容。
- 生成：
  - `containerSequence`
  - `containerCode`
  - 分箱后的数量分配
- 计算：
  - `ifMultiContainers`
  - `code`
  - `msg`

## 5. `lclRules` 的程序语义

这是本程序中最容易被低估的部分。

`lclRules` 的本质不是“给算法增加几个过滤器”，而是“决定商品如何分批进入装箱算法”的上层策略。

程序必须实现以下循环语义：

1. 在当前 group 内，使用当前优先级规则。
2. 用该规则生成一个最大商品子集。
3. 判断该子集是否就是剩余全部商品。
4. 根据标志位决定当前调度目标：
   - 标志位 `0`：当前子集不是最后一批，需要用当前规则产出一个“刚好装不下”的满箱。
   - 标志位 `1`：当前子集已经是全部剩余商品，转入传统多箱装箱流程，把剩余商品完整装完。
5. 对标志位 `0`：
   - 如果当前规则商品子集还能全部装入当前候选箱，说明规则太严格，箱子未满。
   - 此时先判断同规则下是否还有更大子集；若有，则继续扩展子集。
   - 若同规则下没有更大子集，再降级到下一优先级规则。
   - 如果当前规则商品子集无法全部装入当前候选箱，说明该箱达到“刚好装不下”边界，可以锁定该箱装法。
6. 锁定一个箱子后，从当前 group 剩余商品中删除本箱已装商品，然后回到规则编排。
7. 若规则已耗尽，则对当前 group 的所有剩余商品执行兜底装箱，并返回失败状态。

因此，`lclRules` 的实现需要一个独立的规则编排器，而不是散落在若干 if/else 中。

## 6. 错误码和失败语义

根据 `参数列表.md`，失败不只有一种来源：

- 输入非法或关联失败
- 箱型前合法性不满足
- 拼箱规则路径失败
- 多箱限制表达式导致最终失败
- 其他一票否决式非法情况

当前已知的错误码计算包括两类：

- 拼箱规则失败时直接返回 `1`
- 拼箱成功后，根据 `~unpackJudge && ifMultiContainers` 返回成功或失败

这里的精确布尔表达式仍需在设计阶段再做一次程序化整理，但其决策位置已经明确：必须放在结果层，而不是算法层。

## 7. 当前代码现状

从现有代码看，当前项目已经完成接口化重构，当前主路径具备以下能力：

- 纯三维极点装箱核心算法
- 基于接触面积的贪心选点
- JSON 字符串/文件输入 CLI
- 接口级请求模型与响应模型
- 基于 `groupId` 的分组调度
- `csQty` 取余、`stores` 候选箱过滤、`allowDown` 姿态裁剪
- 礼盒/正品/赠品/小样预处理和拼箱规则流程

旧 CSV、可视化和真实订单实验链路已从当前代码中移除，避免与接口主流程混淆。

### 7.1 已具备能力

- `ExtremePointBinPacker`
  - 已实现基础极点法装箱。
- `PackingService`
  - 已串联请求标准化、分组调度、业务规则、结果组装。
- `JsonPackingCli`
  - 已支持从 JSON 文件、JSON 字符串或标准输入读取请求。

### 7.2 尚未具备的关键能力

- 需要继续用更多接口样例覆盖回归。
- 需要补充可审计的调试跟踪输出，用于解释复杂请求在各阶段的数据变化。
- 没有 `(sequence, volume)` 规则驱动的候选箱尝试器。
- 没有 `carrierService=1` 物流快捷路径。
- 没有 `lclRules` 编排器。
- 没有正式的多箱编排器和箱型库存递减机制。
- 没有 `maxWeight` / `capacityRate` 运行时合法性校验器。
- 没有接口级 `code/msg/data.infos` 组装器。

## 8. 核心算法在完整程序中的定位

当前核心算法应该被定位为：

- 输入：
  - 一个已经标准化好的箱体
  - 一组已经经过外围筛选和预处理的待摆放物品
  - 后续还需要支持姿态限制等参数化约束
- 输出：
  - 已摆放物品
  - 未摆放物品
  - 已装体积
  - 利用率

核心算法不应承担以下职责：

- 请求解析
- 商品主数据与订单明细关联
- `csQty` 取余
- 礼盒业务语义判断
- 规则优先级决策
- `groupId` 分组调度
- 多箱循环
- 箱型库存管理
- 运行时合法性校验
- 接口结果回填

## 9. 建议的程序分层

为了支撑完整请求流程，建议程序至少拆成以下层次：

- 接口层
  - 入参解析、出参组装、异常映射
- 标准化层
  - 商品/箱型/订单明细标准化、尺寸缩放、数量取余
- 分组层
  - 按 `groupId` 拆分调度上下文、合并 group 结果
- 约束与预处理层
  - 礼盒逻辑、姿态限制、候选箱过滤
- 编排层
  - 正常快递路径
  - 物流快捷路径
  - `lclRules`
  - 多箱循环
- 核心算法层
  - 极点装箱
- 校验层
  - 重量
  - 容量百分比
- 输出层
  - `infos`
  - `code/msg`

## 10. 已确认决策与剩余待定项

以下内容已经确认：

- `None` 的输入表达是 Java `null`。
- `containerSequence` 按请求级全局递增实现。
- 物流快捷路径不校验 `quantity / maxWeight / capacityRate`。
- 礼盒逻辑不显式验证 `LH` 内部装载可行性。
- `msg` 不需要标准错误码枚举，有明确报错信息即可。
- `capacityRate` 是最大利用率阈值。

以下内容仍需后续确认：

- `SAME_TYPE` 的业务精确定义
