# 程序设计文档

## 1. 文档定位

本文档面向后续 Codex 开发使用，是接口程序重构的核心参照。

事实来源优先级：

1. `参数列表.md`
2. `流程图.md`
3. `分析.md`
4. 当前 Java 代码

当前代码中只有 `src/main/java/com/zeroffa/tdbinpacking/solver/ExtremePointBinPacker.java` 的核心装箱能力可以作为原子操作复用。其它实验流程、CSV 批处理、历史 `bag / box / product` 预处理、真实订单实验、可视化导出等内容，不能作为新接口程序的架构依据。

## 2. 可复用边界

### 2.1 可复用原子能力

可复用能力只有一个：

```java
PackingResult ExtremePointBinPacker.pack(ContainerBox containerBox, List<Item> inputItems)
```

该能力的语义：

- 输入一个确定箱体。
- 输入一组已经离散化、已经预处理、已经筛选过的待装商品。
- 在单个箱体内运行极点法装箱。
- 返回：
  - 已放置商品
  - 未放置商品
  - 已装体积
  - 箱体体积
  - 利用率

### 2.2 不可由核心算法承担的职责

以下职责必须由外围新程序实现，不能塞进 `ExtremePointBinPacker`：

- 请求解析。
- `items / containers / order / lclRules / unpackJudge` 的接口模型。
- `itemCode -> item.code` 关联。
- `quantity % csQty` 取余。
- `groupId` 分组。
- `itemCategory1` 礼盒/正品/赠品/小样语义。
- `allowDown` 姿态限制。
- `stores` 箱型前置过滤。
- `(sequence, volume)` 候选箱排序。
- `quantity` 箱型库存递减。
- `carrierService=1` 物流快捷路径。
- `lclRules` 拼箱规则编排。
- 多箱循环调度。
- `maxWeight / capacityRate` 装后运行时合法性校验。
- `code / msg / data.infos` 出参组装。

### 2.3 核心算法需要的最小适配

核心算法主体复用，但需要围绕它补一个接口适配层：

- 把接口商品转换为算法 `Item`。
- 把接口箱型转换为算法 `ContainerBox`。
- 维护算法 `Item.id` 到订单明细行的映射，用于结果回填。
- 支持 `allowDown=0` 时的姿态限制。
  - 当前 `Orientation.generate(Item)` 总是生成最多 6 个姿态。
  - 新程序需要改造为可传入姿态策略，或在算法输入层提供受限方向枚举。
  - `allowDown=0` 时只允许围绕 z 轴旋转，即最多 2 种姿态。
- 不允许把 `lclRules`、多箱循环、后置校验写入 `ExtremePointBinPacker`。

## 3. 目标程序总流程

一次请求对应一个订单，但一个订单可以包含多个 `groupId`。完整流程如下：

1. 解析 Request。
2. 输入处理：
   - 校验顶层结构。
   - 建立商品和箱型索引。
   - 遍历 `orderDetail`。
   - 关联 `itemCode` 到 `items.code`。
   - 校验尺寸和数量。
   - 执行 `quantity % csQty`。
   - 丢弃散件数量为 0 的行。
   - 聚合调度集合。
   - 离散化所有参与算法的尺寸。
3. 按 `groupId` 分组。
4. 对每个 group 独立运行主流程：
   - `carrierService=1` 走物流快捷路径。
   - `carrierService=0` 走快递路径。
5. 合并所有 group 的合法装箱结果。
6. 计算 `ifMultiContainers`、`code`、`msg`。
7. 展开 `data.infos`。
8. 返回 Response。

## 5. 模块拆分

### 5.1 Facade

`PackingService`

职责：

- 对外提供单入口：

```java
PackingResponse pack(PackingRequest request)
```

- 串联标准化、分组、调度、结果组装。
- 捕获业务异常并转换为 `code=1` 和 `msg`。

### 5.2 标准化器

`RequestNormalizer`

输入：

- `PackingRequest`
- `ScaleConfig`

输出：

- `NormalizedRequest`

职责：

- 顶层空校验。
- 建立 `item.code -> RequestItem` 索引。
- 建立容器标准化列表。
- 遍历 `orderDetail`。
- 执行商品关联。
- 校验尺寸、数量、`allowDown`、`carrierService`。
- 执行 `csQty` 取余。
- 展开散件单位。
- 保留 `groupId`。
- 按 `groupId` 聚合。
- 对 item 和 container 尺寸进行整数离散化。

失败条件：

- `items / containers / order` 为空。
- `itemCode` 无法关联商品。
- 任一参与算法的长宽高小于等于 0。
- `quantity` 非法。
- `csQty` 非法。
- `carrierService` 不是 0 或 1。

### 5.3 group 调度器

`GroupPackingCoordinator`

输入：

- `NormalizedRequest`

输出：

- `RequestPackingPlan`

职责：

- 遍历 `NormalizedRequest.groups`。
- 每个 group 构造独立上下文。
- 调用快递路径或物流路径。
- 合并所有 group 结果。
- 合并失败状态。

要求：

- 不同 group 之间不混装商品。
- 箱型库存是否跨 group 共享，需要作为实现策略明确。默认建议在一次请求内共享 `ContainerCandidate.remainingQuantity`，因为 `quantity` 是一次请求可用库存。

### 5.4 快递路径编排器

`ExpressPackingPlanner`

职责：

- 对单个 group 执行快递路径。
- 先执行商品预处理。
- 生成候选箱集合。
- 调用 `LclRuleOrchestrator`。

输入：

- `PackingGroup`
- `List<ContainerCandidate>`
- `List<LclRuleSpec>`

输出：

- `GroupPackingPlan`

### 5.5 物流快捷路径

`LogisticShortcutPlanner`

职责：

- 对单个 group 执行 `carrierService=1` 路径。
- 按 `stores` 过滤可用箱型。
- 按体积升序选择最小箱型。
- 生成单箱结果。

实现要求：

- 先严格按 `参数列表.md`：直接选符合 `stores` 的最小箱型返回。
- 不检查 `quantity`。
- 不检查 `maxWeight / capacityRate`。
- 不校验所有 group 商品是否实际能放入该箱。

### 5.6 商品预处理器

`ItemPreprocessor`

职责：

- 基于 `itemCategory1` 处理礼盒逻辑。
- 先挑出当前 group 内所有 `LH`。
- 将 `正品 / 小样 / 赠品` 分配到 `LH` 中。
- 分配顺序要求正品优先，然后再分配小样和赠品。
- 分配方式使用轮询，目标是在所有 `LH` 之间尽量均匀。
- 分配后的 `LH` 作为外层 container 调度商品。
- 没有 `LH` 时，所有散件直接进入外层 container 调度。
- 生成核心算法实际需要处理的 `SchedulableItem` 列表。

注意：

- 现有实验代码的 `bag / box / product` 逻辑不复用。
- 第一版不显式验证 `LH` 内部是否装得下被分配商品。
- Response 回填时，被分配到某个 `LH` 的正品、小样、赠品应跟随该 `LH` 所在外箱输出。

### 5.7 候选箱服务

`ContainerCandidateService`

职责：

- 按 `stores` 过滤。
- 按 `(sequence, volume)` 升序排序。
- 检查和扣减 `quantity`。
- 提供“下一候选箱”迭代能力。

`stores` 规则：

- 可用条件为：
  - `orderDetail.stores == container.stores`
  - 或二者任一为 `None`
- 在同一个 group 内，可以认为所有 item 的 `stores` 一致。

### 5.8 传统多箱装箱器

`TraditionalMultiBoxPlanner`

职责：

- 实现图 03。
- 给定一批商品，循环选择箱型并调用核心算法，直到全部装完或失败。

流程：

1. 输入全部剩余商品。
2. 过滤并排序候选箱。
3. 取当前候选箱。
4. 检查 `quantity > 0`。
5. 调用 `CorePackerAdapter`。
6. 若当前待装商品不能全部放入当前箱，尝试下一个候选箱。
7. 若全部放入，执行装后运行时合法性校验。
8. 校验失败则换箱。
9. 校验成功则锁箱、扣减库存、删除已装商品。
10. 若仍有剩余商品，重复。

### 5.9 `lclRules` 编排器

`LclRuleOrchestrator`

职责：

- 实现图 04。
- 它是快递路径主编排器，`lclRules` 不存在启用/关闭分支。

关键状态：

- 当前 group 剩余商品。
- 当前规则。
- 当前规则下已尝试的子集。
- 已锁定箱子列表。
- 是否触发过兜底失败。

流程：

1. 使用当前规则。
2. 按当前规则提取商品子集。
3. 判断该子集是否等于当前 group 剩余全部商品。
4. 若是，`flag=1`：
   - 调用 `TraditionalMultiBoxPlanner`。
   - 成功则返回拼箱成功结果。
   - 失败则返回失败语义。
5. 若否，`flag=0`：
   - 以当前规则商品子集从小到大尝试候选箱。
   - 调用核心算法。
   - 判断当前规则商品子集是否能装满某个箱子。
   - 如果不能装满，说明当前规则太严格，该箱未满。
   - 优先判断同规则下是否还有更大子集。
   - 若有，继续使用当前规则提取更大子集。
   - 若没有，再切换到更低优先级规则。
   - 如果当前规则商品子集未全部装入，说明箱已经装满，锁定本箱装法。
   - 从剩余商品中删除本箱已装商品。
   - 回到当前 group 的规则编排入口。
6. 若 `flag=0` 且无更低优先级规则：
   - 对全部剩余商品按类似 `NO_LIMIT` 方式强制装箱。
   - 复用 `TraditionalMultiBoxPlanner`。
   - 结束后返回失败语义 `code=1`。

规则提取：

- `SAME_SKU_LOT`：`itemCode + batch + lot + manufactureDate`
- `SAME_SKU`：`itemCode`
- `SAME_TYPE`：`itemStyle`
- `SAME_BUILD_ZONE`：`buildCode + zoneCode`
- `SAME_BUILD`：`buildCode`
- `SAME_ZONE`：`zoneCode`
- `NO_LIMIT`：全部剩余商品

### 5.10 核心算法适配器

`CorePackerAdapter`

职责：

- 将 `ContainerCandidate` 转换为 `ContainerBox`。
- 将 `SchedulableItem` 转换为算法 `Item`。
- 调用 `ExtremePointBinPacker.pack(...)`。
- 将 `PackingResult` 转换为业务候选装箱结果。
- 维护 `runtimeId -> SchedulableItem` 映射。
- 支持 `allowDown` 姿态限制。

输出建议：

- `SingleBoxPackingAttempt`
  - `ContainerCandidate container`
  - `List<PackedItem> packedItems`
  - `List<SchedulableItem> unplacedItems`
  - `boolean allPlaced`
  - `long packedVolume`
  - `long containerVolume`
  - `double utilization`

### 5.11 运行时合法性校验器

`RuntimeLegalityValidator`

职责：

- 在候选箱装法锁定前执行。
- 不合法时返回失败原因，由上层继续换箱或降级。

校验项：

- `maxWeight`
  - 非空时，计算当前箱内商品总重量。
  - 超过则不合法。
- `capacityRate`
  - 非空时，计算 `packedVolume / containerVolume`。
  - 超过则不合法。

注意：

- 该模块不组装最终错误码。
- 该模块不直接终止请求。

### 5.12 Response 组装器

`ResponseAssembler`

职责：

- 合并所有 group 的装箱结果。
- 统计箱子数量。
- 计算 `ifMultiContainers`。
- 计算 `code`。
- 生成 `msg`。
- 展开 `infos`。

`code` 规则：

- 初始 `code=0`。
- 如果任一环节存在一票否决失败，`code=1`。
- 如果 `lclRules` 兜底失败语义被触发，`code=1`。
- 否则执行：

```text
code = code || (ifMultiContainers && ~unpackJudge)
```

在 Java 中建议写为：

```java
boolean multiBoxForbiddenFailure = ifMultiContainers == 1 && unpackJudge == 0;
if (multiBoxForbiddenFailure) {
    code = 1;
}
```

## 6. 关键算法语义

### 6.1 `groupId`

- `groupId` 是算法主流程运行边界。
- 一个请求可以包含多个 group。
- 每个 group 独立运行快递或物流主流程。
- 不同 group 的商品不能混装到同一个箱子。
- 所有 group 结果最终合并成一个 Response。

### 6.2 `csQty`

- 如果 `csQty == null`，散件数量等于原始 `quantity`。
- 否则散件数量等于 `quantity % csQty`。
- 散件数量为 0 的商品不进入算法调度集合。

### 6.3 `allowDown`

- `allowDown=1`：允许最多 6 种正交姿态。
- `allowDown=0`：只允许保持向上，只能在 z 轴上旋转，最多 2 种姿态。
- 该规则必须在核心算法枚举姿态时生效。

### 6.4 `stores`

- 箱型可用于某个 group 的条件：
  - group 的 `stores` 与 container 的 `stores` 相同。
  - 或任一方为 `None`。
- `None` 的具体输入表达为 Java `null`。

### 6.5 `lclRules`

- `lclRules` 是快递路径主线，不是可选开关。
- `flag=0` 的目标是产出满箱，不是把当前规则子集全部装完。
- `flag=1` 的目标是把全部剩余商品完整装完。
- 兜底装箱可以继续产生实际装箱结果，但业务状态必须失败。

### 6.6 运行时合法性校验

- 发生在核心算法产出候选装法之后、锁箱之前。
- 不合法则换箱或降级，不在最终 Response 化阶段临时报错。

## 7. 建议开发顺序

第一阶段：接口模型与标准化。

- 新增 Request/Response 模型。
- 新增 `RequestNormalizer`。
- 实现 `csQty`、尺寸缩放、`groupId` 分组。
- 不接入复杂装箱，只验证标准化输出。

第二阶段：核心算法适配器。

- 新增 `CorePackerAdapter`。
- 只支持单箱调用。
- 完成 `runtimeId` 回填。
- 增加 `allowDown` 姿态限制。

第三阶段：传统多箱装箱。

- 实现 `ContainerCandidateService`。
- 实现 `RuntimeLegalityValidator`。
- 实现 `TraditionalMultiBoxPlanner`。
- 能把一个 group 的全部商品装到一个或多个箱子。

第四阶段：`lclRules` 编排。

- 实现规则子集提取。
- 实现 `flag=0` 满箱边界逻辑。
- 实现同规则更大子集。
- 实现规则降级。
- 实现兜底失败语义。

第五阶段：请求级编排。

- 实现 `GroupPackingCoordinator`。
- 接入 `carrierService` 分流。
- 实现物流快捷路径。
- 合并多个 group 结果。

第六阶段：Response 组装。

- 实现 `ResponseAssembler`。
- 实现 `ifMultiContainers` 和 `code`。
- 展开 `infos`。

第七阶段：测试。

- 单 group、单箱成功。
- 单 group、多箱成功。
- 多 group 合并。
- `csQty` 取余。
- `allowDown=0`。
- `stores` 过滤。
- `maxWeight` 换箱。
- `capacityRate` 换箱。
- `carrierService=1`。
- `lclRules` 的 `flag=0`、同规则更大子集、降级、兜底失败。
- `unpackJudge=0` 且多箱返回失败。

## 8. 需要避免的实现错误

- 不要复用实验层 `bag / box / product` 逻辑实现接口礼盒语义。
- 不要把 `lclRules` 写成一个简单过滤器。
- 不要把 `maxWeight / capacityRate` 放在最终 Response 阶段才判断。
- 不要跨 `groupId` 混装商品。
- 不要使用 `item.volume` 字段作为算法事实。
- 不要忽略 `quantity` 箱型库存。
- 不要让 `unpackJudge` 阻止算法运行多箱，它只影响最终状态码。
- 不要让核心算法直接接触接口层对象。

## 9. 已确认决策

- `None` 的具体输入表达是 Java `null`。
- `containerSequence` 使用请求级全局递增。
- 物流快捷路径不校验 `quantity / maxWeight / capacityRate`。
